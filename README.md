## 主流程演示

以下步骤演示一门课预约器材 → 教师领用 → 归还部分 → 提交损耗 → 审批通过 → 周结转的完整流程。

### 步骤 1: 教师预约器材

1. 登录 `zhangsan / 123456`
2. 点击「器材预约」→ 点击「+ 新建预约」
3. 选择课程「大学物理实验」、班级「物理实验A班」、器材「显微镜」、数量 `2`、周次 `2026-W25`
4. 提交 → 预约创建成功，显微镜可用数从 3 减为 1（锁定 2）

### 步骤 2: 实验员审批预约

1. 退出登录，登录 `wangwu / 123456`
2. 在「器材预约」页面，找到状态为「待审批」的预约，点击「审批」
3. 预约状态变为「已审批」

### 步骤 3: 教师领用器材

1. 退出登录，登录 `zhangsan / 123456`
2. 在「器材预约」页面，找到状态为「已审批」的预约，点击「领用」
3. 预约状态变为「已领用」

### 步骤 4: 归还部分器材

1. 仍在「器材预约」页面，点击「归还」
2. 归还数量输入 `1`（部分归还）
3. 预约状态变为「部分归还」，显微镜可用数 +1

### 步骤 5: 申报损耗

1. 点击「损耗申报」→ 点击「+ 申报损耗」
2. 选择关联预约（显微镜预约），损耗数量 `1`，原因填写「镜头摔碎」
3. 提交成功，损耗记录状态为「待审批」

### 步骤 6: 审批损耗

1. 退出登录，登录 `wangwu / 123456`
2. 在「损耗申报」页面，找到待审批记录，点击「通过」
3. 审批通过，显微镜总量从 3 减为 2，锁定数从 1 减为 0

### 步骤 7: 执行周结转

1. 仍在 `wangwu` 账号，点击「周结转」→ 点击「+ 执行周结转」
2. 周次填 `2026-W25`（与预约周次一致），确认
3. 结转成功，展示器材快照、预约汇总、损耗汇总和待归还状态，左上角带「⭐ 最新结转」徽章
4. 点击「📥 导出结转数据」可下载 JSON 文件（文件名 `settlement-2026-W25-settled.json`）

### 步骤 8: 导出→导入结转 JSON（不覆盖业务数据）

**目的**: 验证导出的 JSON 能重新导入恢复结转视图，且不覆盖正式数据

1. 确认步骤 7 已下载 `settlement-2026-W25-settled.json`（如未下载，先点击导出按钮）
2. 在「周结转」页面，点击操作栏的「📤 导入结转 JSON」
3. 先尝试直接导入 `2026-W25` 的 JSON → 系统应提示 **「2026-W25 已存在正式结转记录，导入被拒绝」**（409 冲突，保护正式数据不被覆盖）
4. 为了测试导入流程，先执行「撤销最近结转」（见步骤 9），撤销后再回到此步骤
5. 撤销成功后，2026-W25 的结转记录消失，再点击「📤 导入结转 JSON」
6. 选择之前下载的 JSON 文件，点击「开始导入」
7. 导入成功，结转列表重新出现 `2026-W25` 记录，左上角带 **「📁 导入视图」** 徽章
8. 核对：导入视图的器材快照、预约汇总、损耗汇总、待归还与导出前完全一致（原周次和统计口径保留）
9. 验证业务数据未被影响：查看「器材库存」「器材预约」「损耗申报」页面，数据与导入前一致
10. 点击导入卡片的「🗑️ 移除导入」，可移除导入视图（不影响业务数据）

### 步骤 9: 撤销最近一次结转

**目的**: 验证只能撤销最新周次，撤销后所有结转视图一起回退，且可重复执行结转

1. 以 `wangwu / 123456`（实验员）或 `admin / admin123`（管理员）登录
2. 先确认结转列表有至少一条记录（如 `2026-W25`），且该记录是最新的
3. 在结转卡片的操作区有「↩️ 撤销此结转」按钮（仅最新周次的正式结转才显示此按钮），或直接点击操作栏的「↩️ 撤销最近结转」
4. 弹出确认对话框，列明撤销的影响范围和限制（只能撤销最新）
5. 点击「确认撤销 2026-W25」→ 成功，结转列表中该记录消失
6. 验证重复撤销无效：再次点击「↩️ 撤销最近结转」→ 提示 **「当前没有可撤销的结转记录」**
7. 验证只能撤销最新：先执行 `2026-W25` 的结转，再执行 `2026-W26` 的结转（任意预约周次都可以），此时尝试撤销 `2026-W25` → 提示 **「只能撤销最新周次的结转，当前最新周次为 2026-W26」**
8. 撤销 2026-W26 后，2026-W25 自动成为最新，此时可再次撤销 2026-W25
9. 撤销后，重新执行 `2026-W25` 的结转 → 成功（之前的结转已被软删除，不算重复）
10. 查看「操作日志」：包含 `weekly_settlement`、`revoke_weekly_settlement`、`import_weekly_settlement`、`remove_imported_settlement` 等完整操作链路

### 步骤 10: 结转差异对比 + 说明留痕

**目的**: 对账时对比两周结转差异，并留下可追溯的说明记录

1. 以 `wangwu / 123456`（实验员）或 `admin / admin123`（管理员）登录
2. 先确保系统中至少有 **2 条** 有效结转记录（正式结转或导入视图均可）
   - 如只有 1 条，可分别对 `2026-W24` 和 `2026-W25` 执行两次周结转
   - 或对其中一周导出 JSON，再撤销后重新导入，得到"正式+导入"两条
3. 在「周结转」页面，**每条结转卡片左上角**都会出现「选择对比」复选框
4. 勾选 2 条结转后：
   - 顶部操作栏启用「🔍 差异对比」和「📊 导出对比 CSV」两个按钮
   - 教师角色仅能看到「差异对比」，看不到 CSV 导出按钮
5. 点击「🔍 差异对比」弹出对比对话框，分 4 个板块展示：
   - **📦 器材快照差异**：按「新增 / 减少 / 变化」分类列出总量、可用、锁定的变化
   - **📅 预约汇总差异**：按预约状态（待审批、已审批…）分「新增 / 减少 / 变化」展示条数变化
   - **💥 损耗汇总差异**：申报条数、损耗件数的 A/B 对比及净差异
   - **📤 待归还差异**：按「新增 / 减少 / 变化 + 状态变化」分类展示
   - 所有变化均带有**绿（增）红（减）橙（变）**彩色徽章和 A → B 数字箭头
6. 点击「📊 导出对比 CSV」直接下载文件，格式：
   - 文件名 `settlement-diff-{A周次}_vs_{B周次}.csv`
   - UTF-8 BOM 编码，Excel 直接打开不乱码
   - 列包含「类别 / 变化类型 / 项目 / 字段 / A值 / B值 / 差异」7 列
7. （管理员/实验员）在任意结转卡片点击「➕ 添加结转说明」：
   - 输入对账说明、异常原因、处理计划等内容，保存
   - 卡片顶部出现黄色高亮的「📝 结转说明留痕」区，展示创建人、时间、内容
   - 可「✏️ 修改」或「🗑️ 删除」，所有操作均记入操作日志
8. 教师登录后：
   - 可以查看说明、勾选对比、查看差异
   - **看不到** 添加/修改/删除说明的按钮
   - **看不到** 顶部的「📊 导出对比 CSV」按钮
   - 即使手动调用 API 也会被 403 权限拦截

### 步骤 11: 撤销/移除后的数据收口

**目的**: 验证结转失效后，对比入口、说明、导出记录（含通过 comparison 间接关联的 CSV 导出）一并清理，不留失效数据

1. 准备一条带说明、已被对比、已导出 CSV 的结转（如 `2026-W25`）
2. 以实验员身份对此结转执行「↩️ 撤销此结转」或「🗑️ 移除导入」
3. 撤销成功后验证：
   - 该结转**立即从列表消失**，勾选状态自动解除
   - 顶部操作栏「🔍 差异对比」「📊 导出对比 CSV」按钮自动禁用
   - 再次对相同两条结转执行对比 → 404「结转记录不存在或已撤销」
   - 查看操作日志 `revoke_weekly_settlement` 或 `remove_imported_settlement`，其 `details` 中含：
     - `cleaned_notes`：清理的说明条数
     - `cleaned_comparisons`：涉及该结转（两端任何一端）的对比记录条数
     - `cleaned_exports_single`：通过 `settlement_id` 直接关联（单周导出 JSON）被清理的条数
     - `cleaned_exports_comparison`：通过 `comparison_id` 间接关联（差异对比 CSV 导出）被清理的条数
     - `cleaned_exports_total`：以上两项合计清理的导出总条数（**不再用旧的 `cleaned_exports` 字段**）
   - 调用 `GET /api/settlements/exports` → 已失效周次的任何导出记录**从列表中消失**（接口会 JOIN 结转表查 `revoked=0`，两端都有效才可见）
   - 教师访问此接口 → 固定 403 权限拦截，接口仅对管理员/实验员开放
   - 数据库中 `settlement_notes` / `settlement_comparisons` / `settlement_exports` 的关联记录全部级联清理，**无 orphan 行**

### 步骤 12: 重启后状态保持

**目的**: 验证撤销和导入的状态在重启后仍然正确（基于数据库持久化，不是内存状态）

1. 在完成步骤 11 后，确保当前状态：
   - `2026-W25` 已重新结转（正式记录）
   - 至少有 1 条结转说明留痕
   - 至少有 1 次对比记录 + 1 次 CSV 导出记录（操作日志中能看到）
   - 如有导入记录，记录清楚是哪一周
2. 在终端按 `Ctrl+C` 停止服务器（系统会自动强制落盘）
3. 再次运行 `npm start`
4. 登录后：
   - 结转列表中正式记录和导入记录均保留，徽章正确（⭐ 最新 / 📁 导入）
   - 撤销过的结转**不会**重新出现（软删除标记 `revoked=1` 持久化在数据库）
   - **结转说明在重启后仍然存在**，创建人、创建时间、内容完整保留
   - **说明的编辑历史**（创建/修改时间）仍正确显示
   - **结转导出历史列表（GET /api/settlements/exports）**重启后条数、每条的 filename/row_count/week_key_a/week_key_b 与重启前完全一致
   - **教师访问 /api/settlements/exports 在重启后仍被 403 拦截**
   - 勾选 2 条结转后点击「🔍 差异对比」，结果与重启前完全一致
   - 重新导出 CSV → 行数、列、内容与重启前一致
   - **再次做一次结转→对比→导出→撤销的完整链路**：撤销日志中的 `cleaned_exports_total`、`cleaned_exports_comparison` 字段重启后仍正确计算（不会出现旧实现那样记成 0 的情况）
   - 查看「操作日志」：撤销、导入、**添加/修改/删除说明、对比、导出**的日志全部保留，所有 `revoke_weekly_settlement` / `remove_imported_settlement` 的 details 中包含 5 项 `cleaned_*` 统计字段

### 步骤 13: 导出复盘台账（管理员/实验员专属）

**目的**: 管理员和实验员可以在台账中追溯所有导出操作的完整历史，包括失效记录，支持多维度筛选和台账导出。教师角色无法访问。

1. 登录 `admin / admin123` 或 `wangwu / 123456`（管理员/实验员角色）
2. **前端入口**（两种方式）：
   - 方式一：侧边栏点击「📤 导出历史」或「📒 复盘台账」
   - 方式二：在「周结转」页面操作栏点击「📤 导出历史」或「📒 复盘台账」按钮
   - 教师登录后看不到这两个导航项和按钮
3. 调用台账列表接口 `GET /api/settlements/ledger`，支持以下筛选参数：
   - `week_key_start` / `week_key_end`: 周次范围筛选（如 `week_key_start=2026-W20&week_key_end=2026-W30`）
   - `export_type`: 导出类型筛选（`single`=单周 JSON / `comparison`=对比 CSV）
   - `operator`: 操作人筛选（支持用户名或姓名模糊匹配，如 `operator=admin` 或 `operator=王五`）
   - `invalid`: 是否失效筛选（`false`=有效 / `true`=已失效 / 不传=全部）
   - 示例：`GET /api/settlements/ledger?export_type=single&invalid=false&operator=admin`
4. 列表字段说明：
   - `week_key_a` / `week_key_b`: 来源周次（对比 CSV 有两个周次，单周 JSON 只有 week_key_a）
   - `type`: 导出类型（`single` 或 `comparison`）
   - `filename`: 导出文件名
   - `row_count`: 导出数据行数
   - `related_note_count`: 关联说明条数（实时计算）
   - `last_cleaned_stats`: 最近一次清理统计（包含 `cleaned_notes`、`cleaned_exports_total` 等 5 项，带 `cleaned_` 前缀）
   - `invalid`: 是否失效（0=有效，1=失效）
   - `invalidated_at` / `invalidated_by_user_name` / `invalidated_reason`: 失效时间/失效人姓名/失效原因
   - `created_by_user_name`: 操作人姓名
   - `created_by_username`: 操作人用户名
5. 调用台账详情接口 `GET /api/settlements/ledger/:id`，可看到：
   - `type`: 区分 `single`（普通 JSON）还是 `comparison`（对比 CSV）
   - `comparison_id`: 关联的对比记录 ID（仅对比 CSV 有值）
   - `comparison_diff_summary`: 关联对比的差异摘要（仅对比 CSV 有值）
   - `created_by_user_name` / `created_by_username`: 操作人姓名和用户名
   - `created_at`: 导出时间
6. 调用台账导出接口 `GET /api/settlements/ledger/export/csv`，可导出完整台账 CSV（UTF-8 BOM，Excel 可直接打开）
7. 台账与操作联动验证：
   - **撤销结转**：该周次（作为 A 周或 B 周）关联的所有导出记录自动标记为失效，失效原因为「撤销周结转」，`invalidated_by` 标记操作人
   - **移除导入**：该导入周次关联的所有导出记录自动标记为失效，失效原因为「移除导入结转」
   - **再次导出同一周次**：产生新的有效记录，旧记录保持原状态不变（有效继续有效，失效继续失效）
   - **失效记录收口**：导出历史页面自动过滤失效记录，仅台账页面能看到全部记录
8. 权限验证：
   - 登录 `zhangsan / 123456`（教师角色）
   - 侧边栏不显示「导出历史」和「复盘台账」导航项
   - 周结转页面操作栏不显示「导出历史」和「复盘台账」按钮
   - 调用 `GET /api/settlements/ledger` → 返回 403「权限不足」
   - 调用 `GET /api/settlements/ledger/export/csv` → 返回 403「权限不足」
   - 教师只能看普通结转和差异对比，不能进入台账
9. 重启验证：
   - 停止服务器后重启，台账的筛选结果、统计口径、失效状态、清理统计数据**完全一致**
   - 重启后再次执行撤销/导出操作，联动逻辑正常工作

**台账 CSV 导出字段**（共 19 列）：
`导出ID, 来源周次, 导出类型, 文件格式, 文件名, 行数, 关联说明条数, 状态, 操作人, 导出时间, 失效原因, 失效人, 失效时间, 清理说明数, 清理对比数, 清理单周导出数, 清理对比导出数, 清理总导出数`

## 失败路径演示

### 失败路径 1: 两门课抢同一低库存器材

**目的**: 验证库存冲突检测

1. 登录 `zhangsan / 123456`，预约显微镜 2 个（可用 3 → 锁定 2，可用 1）
2. 退出登录，登录 `lisi / 123456`
3. 尝试预约显微镜 2 个 → 系统提示「库存不足：显微镜可用 1，需 2」，预约被拒绝
4. 在「操作日志」中可看到 `reservation_conflict` 记录

### 失败路径 2: 教师审批自己提交的损耗

**目的**: 验证自我审批禁止

1. 登录 `zhangsan / 123456`，完成预约→审批→领用流程
2. 提交一条损耗申报
3. 退出登录，再以 `zhangsan` 登录（教师角色无审批按钮，此路径需通过 API 模拟）
4. 或用管理员 `admin` 账号尝试审批自己提交的损耗 → 系统提示「不能审批自己提交的损耗申报」
5. 在「操作日志」中可看到 `self_approve_rejected` 记录

### 失败路径 3: 同一周重复执行结转

**目的**: 验证重复结转禁止

1. 登录 `wangwu / 123456`，执行 `2026-W25` 的周结转

## 使用限制和注意事项

### 导出复盘台账限制

1. **角色权限限制**：
   - 仅管理员（admin）和实验员（lab_manager）可访问台账功能
   - 教师角色无台账访问权限，所有台账 API 返回 403
   - 教师可正常使用普通结转、差异对比查看功能，但不能导出台账

2. **数据保留机制**：
   - 台账记录采用**软失效机制**，永不物理删除
   - 撤销结转、移除导入时，关联的导出记录仅标记 `invalid=1`
   - 失效记录保留完整的失效原因、失效时间、失效人、清理统计信息
   - 再次导出同一周次时，产生新的有效记录，旧记录状态保持不变

3. **筛选和统计口径**：
   - 台账列表默认显示全部记录（有效+失效），可通过 `invalid` 参数筛选（`true`/`false`）
   - 周次范围筛选作用于 `week_key_a` 和 `week_key_b`（对比 CSV 的两个周次都会被匹配）
   - `operator` 参数支持用户名或姓名模糊匹配（如 `operator=admin` 或 `operator=王五`）
   - `related_note_count` 为实时计算的关联说明条数，不受失效标记影响
   - `last_cleaned_stats` 存储该记录最后一次被清理时的统计数据（含 `cleaned_*` 前缀，共 5 项）

4. **联动规则**：
   - 撤销结转：该周次（作为 A 周或 B 周）关联的所有导出记录全部失效
   - 移除导入：该导入周次关联的所有导出记录全部失效
   - 再次导出：产生新记录，不影响历史记录状态
   - 所有联动操作均记录操作日志（`revoke_weekly_settlement` / `remove_imported_settlement`），日志中包含 5 项 `cleaned_*` 清理统计

5. **重启一致性保证**：
   - 台账数据完全持久化在 SQLite 数据库中
   - 重启后筛选结果、统计口径、失效状态、清理统计完全一致
   - 重启后联动逻辑正常工作，无状态丢失

### 其他限制

- 撤销只能操作最新周次的正式结转记录
- 导入只能作为视图存在，不能覆盖正式结转数据
- 教师不能审批自己提交的损耗申报

## 技术架构

- **后端**: Node.js + Express
- **数据库**: SQLite (sql.js，纯 JS 实现，数据文件 `data/lab.db`)
- **前端**: 原生 HTML/CSS/JS 单页应用
- **持久化**: 所有写操作即时落盘，重启后数据完整恢复
